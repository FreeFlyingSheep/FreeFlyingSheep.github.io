# COMP5046：自然语言处理（Natural Language Processing）

# Agent

## Agent 概念导入

当 LLM 不再只负责生成文本，而是开始调用工具、读写记忆并与环境交互时，系统形态就从模型扩展为 Agent。

- **Agent**：不仅会生成文本，还会调用工具、使用记忆、与环境交互。
- **Reasoning**：模型在输出前的中间思考过程。
- **Acting**：模型执行外部动作，比如检索、调用 API、操作界面。
- **ReAct**：把推理和行动交替结合的常见范式。

*备注：入门时最容易混淆的是模型能力和系统能力。很多看起来很强的 Agent，强在系统编排，而不只是模型本身。*

这也解释了为什么 Agent 章节会同时讨论推理、行动和基准评测：它们共同决定了系统是否真正可用。

## Agent 思想（Agent Idea）

Agent 是会与世界交互、带记忆（Memory）的模型系统。

Agent 示例：自动驾驶汽车；游戏 AI；聊天机器人；机器人。

语言类 Agent 的历史例子：
纯文本：ELIZA；
世界交互（仍属虚拟）：Siri / Alexa；
机器人控制：SayCan。
这些都更像反射式行为/系统 1（不推理）。

LLM 可作为 Agent，做推理（如 CoT）或行动（如 RAG）。
推理（更新信念）+ 行动（与世界交互）= ReAct。
把推理本身作为模型可用动作之一。
可选：
1. 推理与行动交替；
2. 每步让模型自由选择任意动作。
ReAct 不需要训练，只需 Prompt。

*备注：不需要额外训练并不意味着零成本，提示词设计和工具编排依然会显著影响最终效果。*

那么，把 LLM 放入系统，或让多个 LLM 协作成 Agent 呢？
多个 LLM 相互对话（AutoGen）呢？
多个 LLM 按策略选择（DyLAN）呢？

多模态 Agent 与具身 Agent？
多模态：多输入形态，如文本+图像；
具身：在物理世界有实体，如无人机或机器人。

如何衡量进展？
WebArena；SWE-Bench；PrivacyLens。

**Agent**：可以理解为把 LLM 作为与世界交互系统的一部分。
它结合内部处理与外部交互。

**ReAct**：把推理与行动结合的提示方法，通常效果更好。

**Agent Zoo**：我们看了很多构建 Agent 的方式。
这是非常活跃的研究方向。
核心思想是：LLM 成为系统中的一个组件，而系统可有多种形态。

**基准（Benchmarks）**：
评估 Agent 需要更贴近真实世界的新基准，有时需模拟器让 Agent 主动交互。


## 推理（Reasoning）

什么是推理？

- 人类：心理过程。
- 语言模型：中间生成，即不直接展示给用户的中间文本。
- Agent：内部动作。

方法：

- CoT。
- 类比推理（Analogical reasoning）。
- 自一致性 + CoT（即多输出投票）。
- Reflexion：在 Prompt 中加入对历史输出的反思语句。
- Tree of Thoughts。
- 规划（Planning）。
- 短期 vs 长期记忆：更新 Prompt 是短期记忆；更新数据库是长期记忆。
- 长期记忆实现：读：RAG 风格检索（如 BM25 或 embeddings）；写：把新文本追加为文档。
- DSPy（Declarative Self-improving Python）：定义计算图；由编译器自动优化；可用于结合 Prompt 优化与微调。

什么是推理（Reasoning）？
是模型在生成主要输出前进行思考的任何过程。

**主要方法（Main approaches）**：
CoT 提示是最早方法，很多后续方法是其变体/扩展。
记忆的构建与使用仍是未解难题。
这些方法已经成熟到可以灵活组合（例如结合 DSPy）。

**规划（Planning）**：
我们只简要涉及这个很大的领域。核心信息是：在严格评测下，LLM 在规划上似乎仍有困难。
这也是活跃研究方向。


## 行动（Acting）

- RAG 是与世界行动的一种形式。
- LLM 可通过输出特殊符号来使用工具。
- 若要获取工具使用示例，可在已有数据中寻找工具调用痕迹。
- Web 交互：可用强化学习训练。
- 写代码，例如 OpenHands。
- Anthropic computer use。

什么是行动（Acting）？
是模型超出生成 token本身所做的任何过程。

**常见方式（Main approaches）**：
RAG 是最简单且最常见的行动示例。
对简单 API 工具（计算器、搜索引擎），常可仅靠巧妙提示获得不错效果。
对更复杂交互（如操作整台电脑），目前有大量活跃探索。

**系统编排框架（Orchestration Frameworks）**：
例如 LangChain，可用于组织多步调用、记忆、检索与工具链路。
它的角色更接近应用编排层，而不是单个可被模型直接调用的工具。


# RAG

## RAG 概念导入

RAG 关注的是同一个核心问题：当模型内部知识不够新、不够准或不够全时，如何把外部信息稳定地接入生成过程。

- **RAG**：先检索外部知识，再让模型基于检索结果生成答案。
- **Embedding 检索**：把查询和文档都表示成向量，再比较相似度。
- **上下文注入**：把检索到的片段加入 Prompt，辅助回答。

*备注：RAG 的价值不只是提升正确率，也包括可追溯性。你可以知道答案大致来自哪些文档。*

顺着这个思路，后续内容会从检索形式、相似度计算到文档注入方式逐层展开。

## 检索增强（Retrieval Augmentation）

为什么要访问外部数据？
幻觉（Hallucination）；新事件；稀有信息；错误修正。

**三种通用形式**：

- 并行交互（Parallel Interaction）：K 近邻。kNN-LM。
- 顺序单次交互（Sequential Single Interaction）：基于检索提供示例。
  - 这与示例数量有什么关系？
  - Retrieval ICL：在训练数据中找最相关示例并放入 Prompt。
  - RAG：Retrieval Augmented Generation。REALM。RETRO。
- 顺序多次交互（Sequential Multiple Interaction）：FLARE。

**总体思想**：从文本集合或数据库检索信息，并以某种方式用于生成过程。

**检索（Retrieval）**：最常见做法是计算数据库条目与输入某方面表示向量之间的相似度。

**使用方式（Use）**：多种方案：

- 把检索本身当语言模型并做模型集成；
- 把检索内容放进 Prompt；
- 交互式方法（与 Agent 相关）。

查询 Pinecone 数据库时，查询是如何表示的？
向量 / 嵌入（Embedding）。

在 RAG 系统中，检索文档如何用于响应生成？
文档作为 Prompt 的附加上下文；也可与模型输出结合（类似 kNN-LM 的方式）。

使用 RAG 时，我们算相似度是在比较什么，为什么？
把查询与数据库中每段文本比较，以找出与查询相关的文本。

# MCP

## MCP 概念导入

MCP 处理的是工程接口层问题：如何让模型以统一、可控、可观测的方式连接外部工具和数据。

- **MCP**：一套让模型统一访问工具和数据源的协议化方式。
- **工具调用**：让模型不只回答，还能执行操作。
- **编排**：把提示词、工具、检索和流程组合成稳定系统。

*备注：把 MCP 当成接口层更容易理解。它不替代模型，而是把模型和外部世界连接起来。*

## 工具（Tools）

在 MCP 语境下，Tools 通常指模型可调用的外部能力接口，而不是提示词方法或应用框架。常见类型包括：

- 检索与搜索：查询外部文档、知识库或网页。
- 计算与执行：计算器、代码执行器、结构化函数调用。
- 数据与系统操作：数据库读写、文件操作、业务 API 调用。

这类 Tools 的关键是有明确输入输出契约、权限边界和错误语义，便于系统稳定编排与审计。
